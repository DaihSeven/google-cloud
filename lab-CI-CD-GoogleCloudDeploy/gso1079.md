# 📦 Continuous Delivery com Google Cloud + Google Cloud Deploy

---

## 🧠 1. Conceitos Fundamentais (ANTES do lab)

### 🔁 Continuous Delivery (CD)

É a prática de **automatizar a entrega de software** até ambientes (test, staging, prod), permitindo deploy rápido e seguro.

👉 **Diferença importante:**

- **CI** = build/test  
- **CD** = deploy automático ou semi-automático  

---

### 🚀 Google Cloud Deploy

Serviço que:

- Cria pipelines de deploy  
- Controla promoção entre ambientes  
- Versiona releases  

📌 **Conceitos-chave:**

| Conceito            | O que é                                      |
|--------------------|----------------------------------------------|
| Delivery Pipeline  | Fluxo de deploy (test → staging → prod)      |
| Target             | Ambiente de destino (cluster GKE)            |
| Release            | Versão da aplicação                          |
| Rollout            | Deploy em um target                          |

---

### ☸️ Google Kubernetes Engine (GKE)

- Serviço gerenciado de Kubernetes  
- Onde sua aplicação roda  

---

### 📦 Artifact Registry

- Armazena imagens Docker  
- Tipo um “repositório de builds”  

---

### 🔧 Skaffold

Automatiza:

- build  
- push  
- deploy  

---

Código → Build (Skaffold + Cloud Build)
      → Artifact Registry

⚙️ 3. Passo a Passo Explicado (simplificado)
🔹 Etapa 1 – Variáveis
export PROJECT_ID=$(gcloud config get-value project)
export REGION="Region"

📌 Define:

projeto ativo
região dos recursos
🔹 Etapa 2 – Criar clusters
gcloud container clusters create test ...

Cria 3 ambientes:

test
staging
prod

💡 Simula ambientes reais de empresa

🔹 Etapa 3 – Criar repositório de imagens
gcloud artifacts repositories create web-app

👉 Onde suas imagens Docker serão armazenadas

🔹 Etapa 4 – Build da aplicação
skaffold build ...

💡 O que acontece aqui:

Lê o skaffold.yaml
Builda imagens Docker
Envia para Artifact Registry
📄 Entendendo o skaffold.yaml

Ele define:

🧱 build → imagens
🚀 deploy → manifests Kubernetes
🔹 Etapa 5 – Criar pipeline
gcloud beta deploy apply --file=delivery-pipeline.yaml

Pipeline definido como:

test → staging → prod
🔹 Etapa 6 – Targets

Cada target aponta para um cluster:

name: test
gke:
  cluster: ...

💡 Especial:

prod exige aprovação manual
🔹 Etapa 7 – Criar release
gcloud beta deploy releases create web-app-001

📌 Isso:

versiona a aplicação
inicia deploy automaticamente em test
🔹 Etapa 8 – Promover para staging
gcloud beta deploy releases promote

👉 Move para o próximo ambiente

🔹 Etapa 9 – Promover para prod (com aprovação)
gcloud beta deploy rollouts approve ...

📌 Segurança:

evita deploy acidental em produção
⚠️ 4. Erros Comuns + Soluções
❌ 1. Cluster não está RUNNING

Erro:

status: PROVISIONING

✔️ Solução:

esperar alguns minutos
rodar novamente:
gcloud container clusters list
❌ 2. Erro no Skaffold build

Possíveis causas:

API não ativada
bucket não criado

✔️ Solução:

gcloud services enable cloudbuild.googleapis.com
gsutil mb -p $PROJECT_ID gs://${PROJECT_ID}_cloudbuild
❌ 3. Imagens não aparecem no Artifact Registry

✔️ Verifique:

gcloud artifacts docker images list ...

Se estiver vazio:

build falhou
repositório incorreto
❌ 4. Pipeline não encontra targets

Erro:

Unable to get target test

✔️ Solução:

gcloud beta deploy apply --file target-test.yaml
❌ 5. Rollout travado em PENDING_APPROVAL

✔️ Solução:

gcloud beta deploy rollouts approve ...
❌ 6. Kubernetes sem pods
kubectl get all -n web-app

Se estiver vazio:

✔️ Verifique:

namespace criado
contexto correto
kubectx test
❌ 7. Erro de permissão

✔️ Solução:

gcloud services enable clouddeploy.googleapis.com
🧩 5. Resumo Final (cola pra prova)
Cloud Deploy gerencia CD
Pipeline = sequência de ambientes
Release = versão da aplicação
Targets = clusters (GKE)
Skaffold = build + deploy
Artifact Registry = imagens

Build → Registry → Release → Test → Staging → Prod
